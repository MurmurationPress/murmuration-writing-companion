# Manuscript Part creation

The Manuscript navigator creates a Part as one ordinary authoritative Markdown note. The owning Book is resolved only from `ManuscriptBookSelectionService`; active-note navigation is not Book authority.

## Contract

```yaml
---
type: part
title: "Author-supplied title"
parent: "[[full/vault-relative/book/path]]"
manuscript_order_key: "GENERATEDKEY"
---
```

The parent is the full vault-relative Book path without `.md`. The key is the existing ten-character uppercase base-36 distributed key. No legacy `manuscript_order` array, heading, body, Scene, editorial data, Story World data, compiler data or publishing data is created.

## Three independent choices

- **Filesystem location** organises the note and is editable convenience only.
- **parent** establishes authoritative containment directly beneath the selected Book.
- **manuscript_order_key** establishes authoritative position among that Book's direct children.

The proposed location prefers the common folder of existing Parts in the Book, then the established associated Book folder, then the common folder of direct children, then the Book note's parent. None of those folder conventions is structural authority.

## Placement and safety

The placement selector exposes one option for each direct-child boundary and labels neighbours as Part or Scene. Keys come only from `manuscriptOrderKeyBetween`. Existing children are never moved, reparented, re-keyed or rewritten. Empty distributed books are supported; legacy, mixed, malformed, duplicate-key, unsafe and exhausted-key structures block with reconciliation guidance.

Confirmation re-resolves the selected Book and the specific insertion boundary. It verifies the adjacent paths, kinds, parents and keys and regenerates the same key. Unrelated changes elsewhere do not stale a safe boundary. Destination, duplicate-title and folder assumptions are also checked before and after previewed folder creation.

## Creation and recognition

The shared #140 seam performs request coalescing, explicit folder creation, one-file creation, exact-byte read-back, guarded cleanup and bounded recognition. Recognition additionally verifies Part kind, direct parent, key and adjacent position. Delayed or structurally incorrect projection preserves the verified note and offers **Open note**; it never rewrites neighbours.

After successful recognition, `bookPath` remains the owning Book and `contextPath` temporarily reveals the new Part. Context is presentation state, not manuscript authority or a duplicate Part record. Continuity Review remains scoped to the Book.

## Real-vault verification

The completed FEVER smoke test verified the full workflow against the provisional Book 4. FEVER was created as the first Part of an empty Book using the exact path-qualified Book wikilink shown in the preview (the full vault-relative Book note path, without `.md`) and the canonical ten-character uppercase base-36 key generated by the existing distributed-order allocator. The previewed parent and key matched the created note and remained authoritative after restart.

Book 4 remained the selected manuscript scope while FEVER became the transient navigator `contextPath`; Continuity Review continued to target Book 4. Disposable structures verified beginning, middle and end insertion without changing the bytes of any existing sibling. Stale insertion boundaries were rejected without a write, as were unsafe and exhausted ordering structures.

The test created no Scene and wrote no editorial, Story World, compiler or publishing data. Filesystem location remained an organisational choice only: the `parent` wikilink established containment and `manuscript_order_key` established position.

## Boundaries

Issue #142 remains responsible for Scene creation and optional scene metadata. Issue #143 remains responsible for detaching a Scene without deleting its note. This workflow does not implement nested Parts, Scene movement, Part deletion, renaming, later reordering, templates, broad key rebalance, Story World authoring, or publishing/compiler setup.
